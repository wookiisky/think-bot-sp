# 侧边栏核心回归

## 1. 关联能力

- 侧边栏打开
- 页面提取
- 文本/图片发送
- 快捷输入
- 消息编辑与分支操作
- 分支预览
- 顶部控制区
- 提取区与输入区拖拽
- 清空与导出

## 2. 核心职责清单

- 在普通网页打开 side panel。
- 展示缓存或完成提取。
- 提取内容区常驻显示，且与当前 `promptTab` 聊天区独立并存。
- 支持顶部控制区复制、重新提取、页面级清空和页面跳转按钮。
- 支持直接提问和快捷输入。
- 支持图片预览添加与移除。
- 正确展示主回答和分支。
- 支持编辑用户消息、继续新增分支、停止/删除目标分支。
- 支持提取区和输入区高度拖拽。
- 支持当前 `promptTab` 停止、清空、导出。

## 3. 验证点

- 正常流：
  - 打开侧边栏并显示内容。
  - 阶段 3 必须覆盖“两阶段 bootstrap -> 黑名单确认 -> 正文提取成功”闭环。
  - 顶部控制区、提取内容区、`promptTab` 区、聊天区、输入区的上下结构稳定。
  - 提取内容区在切换 `Chat` 和快捷输入 `promptTab` 后仍保持常驻可见。
  - side panel 首屏初始化采用主动拉取 bootstrap，首屏不依赖 background 主动推送命令。
  - `promptTab` 数量较多时支持多行换行，激活 `promptTab` 和状态标记仍可辨认。
  - 顶部控制区复制提取内容、打开设置页、打开历史页、打开 GitHub 正常。
  - 切换提取方式并刷新。
  - 提取区高度拖拽后布局稳定。
  - side panel 再次打开时，已有缓存不重复提取。
  - 文本发送、图片发送成功。
  - 图片预览移除后不会误发送。
  - 自动触发快捷输入在后台完成。
  - side panel 再次打开时，已有历史或 loading 的 `promptTab` 不重复自动触发。
  - 编辑用户消息后重发成功且旧结果被裁剪。
  - 继续新增分支只追加目标消息的分支。
  - 顶部页面级清空后，页面缓存、页面状态和 `promptTab` 会话一并清除。
  - 底部当前 `promptTab` 清空后，只清除当前 `promptTab` 会话，不影响其他 `promptTab` 和页面提取内容。
- 错误流：
  - content script 未连接。
  - 提取失败。
  - 流式中断。
- 异常流：
  - side panel 关闭重开恢复 loading。
  - `browserTab A` 打开 side panel，切到 `browserTab B` 自动隐藏，切回 `browserTab A` 不自动展示，但再次点击时应可直接打开。
  - 用户取消请求。
  - content script 断连后自动刷新一次再重试。
  - 静态 content script 未挂载时，background 能先按需注入再继续采集。

阶段 3 当前复核现状：

- 已覆盖自动化：两阶段 bootstrap、黑名单确认后提取、普通网页入口、受限页退化、`browserTab` 切换隐藏后不自动展示且当前活动页重新预配置。
- 已知问题：无。

阶段 4 当前复核现状：

- 已覆盖自动化：`Chat` 主回答发送、首包流式、完成写历史、side panel 重开恢复、worker 重启后恢复 loading、空会话导出守卫。
- 仍未覆盖：快捷输入多标签真实工作台、分支并发、消息编辑、重试。

阶段 3 本次修复复盘：

- 第一轮修复只补了 `chrome.storage.session` 运行态，解决了 worker 重启后旧 tab 清理，但没有彻底解释真实浏览器中的自动恢复问题。
- 继续排查后确认，WXT 保留入口名 `sidepanel.html` 会导致构建产物带上全局 `side_panel.default_path`，这是“切换后仍会恢复”的结构性根因之一。
- 路由改成 `sidebar.html` 后，`entry-shell` 与 sender 校验、E2E query 参数、文档都必须一起调整，否则会出现测试绿但真实入口错位的假象。
- 第二轮修复又暴露出“需要点击两次才能打开”，最终通过活动 tab 预配置 side panel 收口，测试口径也从“不自动恢复”细化成“不自动展示，但下一次点击应可直接打开”。

## 4. 关键边界条件

- 黑名单命中。
- 黑名单命中时先展示确认层，未确认前不提取、不自动触发。
- 受限页面退化。
- 空内容页面。
- `promptTab` 名称较长且数量较多。
- 快捷输入已有历史时防重复触发。
- 快捷输入仍在执行中时防重复触发。
- 清空当前 `promptTab` 后再次进入页面可重新自动触发。
- 页面级清空后重新进入页面可重新提取并重新自动触发。
- 输入区高度较大时按钮布局仍可操作。

## 5. 历史高风险场景

- `promptTab` loading 与 `has-content` 状态错乱。
- 分支停止误影响其他分支。
- 顶部按钮操作导致当前输入内容丢失。
- 页面级清空与 `promptTab` 级清空语义混淆。
- 清空时未先取消进行中请求。
- 编辑用户消息后保留了过期回答。
- 黑名单确认与提取链路竞态，导致先提取后拦截。

## 6. 测试数据与环境

- 普通长文页
- 搜索结果页
- 受限页
- 至少 1 个完整可用模型
- 至少 1 条 autoTrigger 快捷输入

## 7. 阶段 2.5 入口壳层验收

- 阶段 2.5 先验证 side panel 入口壳层基线，不把真实提取与聊天工作台当作已完成能力。
- `tests/e2e/entry-shell.spec.ts` 当前应验证 `sidebar.html` 可直接打开，并进入 `sidebar-shell` 的稳定态；不要再依赖瞬态 loading 或旧的 `page-shell` 占位结构。
- Playwright E2E 必须先构建当前源码，再加载 `.output/chrome-mv3`，避免读取陈旧扩展产物导致壳层回归失真。
