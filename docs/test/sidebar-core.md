# 侧边栏核心回归

## 1. 关联能力

- 侧边栏打开
- 页面提取
- 文本/图片发送
- 快捷输入
- 消息编辑与分支操作
- 分支阅读列
- 顶部控制区
- 提取区与输入区拖拽
- 清空与导出

## 2. 核心职责清单

- 在普通网页打开 side panel。
- 展示缓存或完成提取。
- 提取内容区常驻显示，且与当前 `promptTab` 聊天区独立并存。
- 支持顶部控制区复制、重新提取、页面级清空和页面跳转按钮。
- 支持直接提问和快捷输入。
- 快捷输入不再预填 prompt 草稿，首次进入按规则直接触发。
- 支持图片预览添加与移除。
- 正确展示主回答和分支。
- 支持编辑用户消息、用户消息重试、助手分支重试、主分支切换、继续新增分支、停止/删除目标分支。
- 支持提取区和输入区高度拖拽。
- 支持当前 `promptTab` 停止、清空、导出。
- 支持 icon button 的即时 tooltip 与删除/清空类 mini confirm。
- 支持 `Readability -> Markdown` 的原始纯文本展示。
- 支持 tab 纯视觉状态层和消息 hover 工具条。

## 3. 验证点

- 正常流：
  - 打开侧边栏并显示内容。
  - 阶段 3 必须覆盖“两阶段 bootstrap -> 黑名单确认 -> 正文提取成功”闭环。
  - 首次安装后默认快捷输入会与 `Chat` 一起出现在 `promptTab` 区。
  - 快捷输入 `promptTab` 首次进入时，若当前无历史、无 loading，应直接发送真实 prompt，而不是把 prompt 放进输入框。
  - 顶部控制区、提取内容区、`promptTab` 区、聊天区、输入区的上下结构稳定。
  - 顶部不再展示单独 `browserTab` 标签行。
  - 提取内容区在切换 `Chat` 和快捷输入 `promptTab` 后仍保持常驻可见。
  - `Readability` 正文按原始 Markdown 纯文本展示，标题、列表等语法不再渲染成富文本结构。
  - side panel 首屏初始化采用主动拉取 bootstrap，首屏不依赖 background 主动推送命令。
  - `promptTab` 数量较多时支持多行换行，标签宽度按名称自适应，激活 `promptTab` 和状态标记仍可辨认。
  - 已有文本的 `promptTab` 底部显示彩色线，loading 的 `promptTab` 显示彩色边框流动效果，且两者都不改变按钮尺寸。
  - 顶部控制区复制提取内容、打开设置页、打开历史页、打开 GitHub 正常；页面级反馈通过 toast 展示。
  - icon button hover / focus 时立即展示 tooltip，不依赖原生 `title`。
  - 切换提取方式并刷新。
  - 提取区高度拖拽后布局稳定。
  - side panel 再次打开时，已有缓存不重复提取。
  - 文本发送、图片发送成功。
  - `Enter` 发送、`Shift + Enter` 换行、composition 期间不误发送。
  - 图片预览移除后不会误发送。
  - 自动触发快捷输入在后台完成。
  - 页面已有缓存正文时，side panel 打开后仍会补触发已配置 `autoTrigger` 的快捷输入。
  - side panel 再次打开时，已有历史或 loading 的 `promptTab` 不重复自动触发。
  - 自动触发产生的用户消息默认展示快捷输入名称，进入编辑态时展示真实 prompt，编辑保存后展示编辑后的文本。
  - 编辑用户消息后重发成功且旧结果被裁剪。
- 用户消息重试后会裁剪其后的消息，并重建新的助手消息。
  - 消息正文支持 Markdown 渲染，复制 Markdown 与复制纯文本入口可见。
  - 消息正文不再显示单独角色行，也不再使用带边框卡片，改为横线分隔。
  - 助手生成中只显示消息开头 loading 圈，停止按钮显示在该 loading 消息内部。
  - 用户和助手消息按钮在完成后显示在消息内部右上角，鼠标移入当前消息再显示，助手按钮纵向排列。
  - 继续新增分支先弹出模型列表；选中模型后只追加目标消息的一个分支。
  - 手动新增分支允许重复选择同一模型，UI 会显示 `模型名 #1/#2/...`。
- 多分支时展示响应式并行多列，列最小宽度固定为 350px。
- 当前轮可切换“后续继续对话”的主分支，后续已有消息时按钮禁用。
- 点击快捷输入时若提取区为空，会先自动提取再触发发送。
  - 顶部页面级清空后，页面缓存、页面状态和 `promptTab` 会话一并清除。
  - 底部当前 `promptTab` 清空后，只清除当前 `promptTab` 会话，不影响其他 `promptTab` 和页面提取内容。
  - 页面清空、标签清空、分支删除、图片移除在首次点击时只展开 mini confirm，不直接执行副作用。
- 错误流：
  - content script 未连接。
  - 提取失败。
  - 流式中断。
- 异常流：
  - side panel 关闭重开恢复 loading。
  - `browserTab A` 打开 side panel，切到 `browserTab B` 自动隐藏，切回 `browserTab A` 不自动展示，但再次点击时应可直接打开。
  - 用户取消请求。
  - content script 断连后自动刷新一次再重试。
  - 静态 content script 未挂载时，background 能先按需注入再继续采集。

## 4. 关键边界条件

- 黑名单命中。
- Google / Bing / 百度搜索结果页默认命中黑名单。
- 黑名单命中时先展示确认层，未确认前不提取、不自动触发。
- 受限页面退化。
- 空内容页面。
- `promptTab` 名称较长且数量较多。
- 快捷输入已有历史时防重复触发。
- 快捷输入仍在执行中时防重复触发。
- 清空当前 `promptTab` 后再次进入页面可重新自动触发。
- 页面级清空后重新进入页面可重新提取并重新自动触发。
- 输入区高度较大时按钮布局仍可操作。

## 5. 历史高风险场景

- `promptTab` loading 与 `has-content` 状态错乱。
- 分支停止误影响其他分支。
- 顶部按钮操作导致当前输入内容丢失。
- 页面级清空与 `promptTab` 级清空语义混淆。
- 清空时未先取消进行中请求。
- 编辑用户消息后保留了过期回答。
- 黑名单确认与提取链路竞态，导致先提取后拦截。
