# 模型调度与流式回归

## 1. 关联能力

- Provider 解析
- 主回答流式
- 分支并发
- 消息编辑与分支扩展
- 取消与错误回收
- loading 恢复

## 2. 核心职责清单

- 正确解析 4 类 Provider。
- 阶段 4 当前先保证主回答可独立流式输出。
- 图片能力校验正确。
- 用户消息编辑后只重建受影响结果。
- side panel 重开后恢复 loading UI。
- 空会话导出必须明确失败，不能产出空 Markdown 文件。

## 3. 验证点

- 正常流：
  - `streamText` 输出主回答。
  - 事件顺序固定为 `STARTED -> CHUNK* -> FINISHED`。
  - 快捷输入触发多分支。
  - 分支完成后写入历史。
  - 编辑用户消息后旧回答被裁剪并生成新回答。
  - 对已有回答继续新增分支不覆盖旧分支。
  - 有效会话导出成功，包含主回答与分支结构。
- 错误流：
  - Provider 认证失败。
  - 不支持图片的模型收到图片输入。
  - setup 在写入助手占位后失败时，助手消息被补偿为 `error`。
  - 某个分支返回错误。
  - 非法目标消息触发编辑请求。
  - 空会话触发导出请求时返回明确失败。
- 异常流：
  - 用户取消。
  - service worker 重启后恢复。
  - port 断开后重新订阅。
  - port 推送失败不覆盖已持久化结果。
  - loading 清理失败不覆盖已完成的主生命周期结果。
  - side panel 关闭后重新打开，通过持久化 loading 恢复到正确消息锚点。

## 4. 关键边界条件

- 空文本但有图片。
- 多分支同时完成或同时错误。
- 同一消息多次重试产生多分支。
- 主回答为空且仅存在失败分支时禁止导出空文件。

## 5. 必须长期回归的历史风险

- chunk 顺序错乱。
- branch loading 长期残留。
- 取消只停主回答未停分支。
- 已终态助手消息被重复追加或被二次覆盖。
- 编辑消息后仍残留旧分支或旧回答。
- 空导出请求仍生成了空文件，导致用户误以为导出成功。
