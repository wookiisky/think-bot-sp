# 代码仓库文档系统规范

## 1. 目标

当代码仓库变大后，AI coding agent 很难在有限上下文内快速拿到正确约束，人也无法长期记住所有实现细节。这套文档系统的目标是提供一个稳定抽象层，用来：

- 帮助 agent 快速定位最相关的模块、数据域、流程和限制条件。
- 帮助人理解系统结构、管理模块边界、评估变更影响。
- 把“职责、边界、流程、约束、测试标准”从隐性知识变成可维护的显性知识。

这套文档不替代代码。它负责描述代码背后的稳定事实和验证标准。

## 2. 基本原则

- 抽象优先：写模块职责、边界、约束、流程和决策，不转写代码。
- 检索优先：先让人和 agent 找到正确文档，再决定是否深入代码，表达精炼、直白、通俗易懂。
- 单一职责：每个文档只承载一个稳定主题。
- 主来源唯一：同一事实只能有一个主文档，其他文档只引用。
- 可验证：文档中的关键行为和约束，必须能映射到代码、配置、测试或决策记录。
- 与代码同步：模块职责、存储结构、流程、错误处理变化时，文档必须同步更新。

## 3. 整体约束

- 只写已实现事实、已确认决策或明确声明的目标状态，确定性表达。
- 未确认内容必须标记为“待确认”。
- 跨模块内容写入 `flow.md` 或 `decision_log.md`，不要塞进单模块文档。
- 文档命名必须使用代码中的真实术语，方便搜索映射。
- 长文档必须使用固定章节，避免自由散文式写法。
- 文档必须优先描述“不能破坏什么”和“如何验证没有被破坏”。

## 4. 顶层结构

文档系统顶层为 `Docs/`。

```text
Docs/
  index.md
  app.md
  tech_stack.md
  decision_log.md
  flow.md
  DataSchema/
    <domain-or-table>.md
  dao/    
    <module>.md  
  test/
    <module>.md
  <BigModuleA>/
    <module>.md
  <BigModuleB>/
    ...
```

说明：

- 顶层第一级目录代表系统的稳定部分，不是代码物理目录镜像。
- `DataSchema/`、`dao/`、`test/` 与各业务模块同级，分别提供数据视角、数据访问层、测试模块层。
- `index.md`、`app.md`、`tech_stack.md`、`decision_log.md`、`flow.md` 直接位于根目录。

`<BigModuleA>` 代表其他大模块划分，比如通用工具，基于业务的划分等。

## 5. 顶层文件职责

### 5.1 `index.md`

用途：整个文档系统入口。

必须包含：

- 所有一级目录和关键文档索引。
- 每个文档的简短说明。
- 常见任务到文档入口的映射。

要求：

- 只做导航，不承载细节。

### 5.2 `app.md`

用途：项目总览。

必须包含：

- 项目目标和核心业务范围。
- 启动入口和主要运行单元。
- 整体架构和主要模块关系。
- 关键运行约束和主要外部依赖。

### 5.3 `tech_stack.md`

用途：技术选型说明。

必须包含：

- 语言、框架、运行时。
- 核心基础库及用途。
- 数据库、缓存、消息系统、外部服务。
- 构建、部署、测试、监控相关组件。
- 关键版本和兼容性约束。

### 5.4 `decision_log.md`

用途：记录关键需求变化和技术决策。

每条记录必须包含：

- 日期。
- 背景。
- 决策。
- 原因。
- 影响范围。
- 放弃的备选方案。
- 后续需要同步的模块或文档。

### 5.5 `flow.md`

用途：描述跨模块流程。

必须包含：

- 主流程。
- 错误流和异常流。
- 涉及模块及职责分工。
- 状态变化。
- 重试、补偿、回滚、降级等恢复策略。
- 关键验证点。

## 6. `DataSchema/` 规范

`DataSchema/` 下每个文件代表一个稳定数据域。它可以是：

- 数据库表。
- 一组强关联表。
- Redis domain。
- 缓存命名空间。
- 消息持久化结构。
- 文件或对象存储域。

每个文件必须包含：

### 6.1 数据域概述

- 名称。
- 存储类型。
- 业务意义。
- 所属模块。
- 上下游依赖。

### 6.2 字段定义

每个关键字段至少说明：

- 字段名。
- 类型。
- 是否必填。
- 默认值。
- 含义。
- 取值约束或枚举说明。

### 6.3 索引与限制

- 主键、唯一键、联合索引。
- 分区键或路由键。
- TTL、归档、删除策略。
- 外键或逻辑关联。
- 幂等键、去重键等业务约束。

### 6.4 读写路径

- 谁读。
- 谁写。
- 典型查询方式。
- 典型更新方式。
- 并发和一致性要求。

### 6.5 生命周期与风险

- 创建、更新、删除时机。
- 迁移或兼容策略。
- 脏数据、并发冲突、缓存一致性、热点等风险。

### 6.6 测试要求

必须突出“这个数据域怎样被证明是安全的”。至少包含：

- 字段约束测试。
- 索引和唯一性约束测试。
- 并发写入/重复写入测试。
- 读写一致性测试。
- 删除、过期、迁移、兼容性测试。

## 7. 业务模块目录规范

一级业务模块目录只表达业务能力边界，例如 `User`、`Billing`、`Conversation`、`TaskOrchestration`。这些目录与 `DataSchema/`、`dao/`、`integration/`、`test/` 处于同一层级。

### 7.1 模块主文档

每个模块文件必须回答以下问题：

- 这是什么模块。
- 它负责什么。
- 它不负责什么。
- 它依赖什么。
- 它如何工作。
- 它出错时如何处理。
- 修改它时哪些行为不能被破坏。
- 如何证明它仍然正确。

建议固定结构如下：

```md
# <模块名>

## 1. 模块定位和核心抽象
## 2. 能力边界
## 3. 对外接口
## 4. 关键流程
## 5. 错误与异常处理
## 6. 数据与状态
## 7. 约束与禁止事项
## 8. 测试要求
```

其中 `测试要求` 必须直接覆盖模块职责和行为，而不是只列测试类型。至少要写清：

- 职责测试：证明模块确实完成了它承诺的核心能力。
- 正常流测试：输入合法时的正确行为。
- 错误流测试：输入非法、依赖失败、状态不满足时的行为。
- 异常流测试：超时、中断、重试、幂等、并发、部分失败时的行为。
- 不变量测试：关键状态、顺序约束、一致性约束是否被保持。
- 可观测性测试：关键日志、指标、告警、错误码是否符合约定。

原则：

- 测试章节必须围绕“职责和行为”组织，而不是按单测/集测简单分类。
- 每条关键职责至少要有一个正向验证点和一个反向验证点。
- 每条禁止事项最好有对应的保护性测试或校验点。

### 7.2 顶层 `dao/` 目录

用途：从全系统的数据访问视角记录访问抽象，而不是挂在单个业务模块下面。

每个文件建议包含：

- 关联哪个业务能力或模块。
- 访问哪些数据域。
- 提供哪些查询和写入能力。
- 事务边界、并发约束、缓存参与方式。
- 上层可以依赖什么，不能依赖什么。
- 需要验证的查询正确性、写入幂等性、一致性和性能边界。

要求：

- 文件可以按业务能力命名，例如 `conversation-history.md`、`billing-ledger.md`。
- 具体数据域定义仍以 `DataSchema/` 为主来源，`dao/` 只描述访问方式和约束。

### 7.3 顶层 `test/` 目录

用途：从全系统验证视角记录“哪些职责和行为必须被证明正确”。

每个文件建议包含：

- 关联哪个业务能力、模块或流程。
- 核心职责清单。
- 每项职责对应的验证点。
- 正常流、错误流、异常流矩阵。
- 关键边界条件。
- 必须长期回归的历史缺陷或高风险场景。
- 测试数据、夹具、模拟依赖和环境要求。

要求：

- `test/` 目录不是测试代码目录说明，而是验证规范。
- 文件可以按业务能力或关键流程命名，例如 `conversation-send-message.md`、`billing-subscription.md`、`user-login.md`。
- 如果验证对象是跨模块流程，应优先与 `flow.md` 建立链接。
