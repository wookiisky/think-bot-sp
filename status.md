# Think Bot 当前实现对照状态

## 1. 结论先行

当前仓库处于“重写进行中”的中间状态，不是 `prd_docs/product-functional-spec.md` 所描述的完整产品状态，也不是 `/docs` 顶层文档描述的完整目标状态。

按代码事实判断，当前已形成闭环的能力主要只有 4 块：

1. 浏览器入口与受限页退化。
2. 侧边栏首屏 bootstrap、黑名单阻断、正文提取。
3. 单一 `Chat` 标签页的主聊天流、流式恢复、停止。
4. 设置页的基础本地配置闭环。

缺口主要集中在：

1. 对话管理页仍是占位页。
2. 快捷输入多标签、自动触发、分支并发、历史恢复、消息编辑、分支操作均未落地。
3. 黑名单设置编辑、会话导出、页面级/标签级清理等关键能力未落地；同步目前只完成配置级手动闭环。
4. 多份文档把“目标能力”写成了“当前已实现能力”，与实际代码明显漂移。

## 2. 判定标准

- 只以当前代码可证明的行为为准，不以文档宣称为准。
- “已实现”要求存在前后端链路或至少存在完整本地闭环。
- “缺失”表示规格或方案明确要求，但代码中没有对应实现。
- “未对齐”表示代码有部分实现，但行为、范围或完成度与文档描述不一致。

## 3. 对照 `prd_docs/product-functional-spec.md`

### 3.1 已实现

- Chrome MV3 扩展基础骨架已存在，包含 `background`、`content script`、`sidebar`、`options`、`conversations`、`welcome` 入口。
- 点击扩展图标时，普通网页会为当前 tab 启用 side panel；受限页会退化打开 `conversations` 页。
- 扩展图标右键菜单可直接打开 `conversations` 页。
- 首次安装会根据浏览器语言打开内部 `welcome` 页，并带上 `locale` 参数。
- 侧边栏会主动请求 `GET_SIDEBAR_BOOTSTRAP`，先恢复页面缓存、会话和 loading 状态。
- 黑名单命中后会先阻断侧边栏工作流，用户确认后才继续提取。
- 页面提取链路已存在：优先 Readability，失败后回退 Jina，并把结果写入页面缓存。
- 侧边栏已具备单一 `Chat` 标签的消息线程、模型选择、文本输入、图片文件选择、发送、停止。
- 主聊天流已形成后台闭环：写用户消息、写助手占位、流式 chunk 持久化、结束/失败/取消收敛、loading 清理。
- side panel 重开后可通过持久化的 loading state 恢复进行中的主回答。
- 设置页已实现配置读取、保存、恢复默认、导入、导出、本地缓存统计与清理。
- 设置页已支持左侧 5 栏导航、顶部动作区、基础设置草稿保留，以及语言和主题即时预览。
- 设置页已支持标签页快捷输入新增、编辑、软删除、顺序调整，以及语言模型新增、复制、软删除、顺序调整和单项展开编辑。
- 设置页已支持同步配置编辑、连接测试、手动同步和最近同步时间回写。

### 3.2 缺失

- 对话管理页历史工作台未实现，当前仍是占位页，缺少页面列表、搜索、恢复、继续对话、标题编辑、删除。
- 侧边栏快捷输入多标签未实现，只有固定 `Chat` 标签。
- 快捷输入自动触发未实现。
- 多模型分支并发回答未实现。
- 分支停止、删除、继续新增未实现。
- 用户消息编辑、重试未实现。
- 页面级清空与 `promptTab` 级清空未实现。
- 会话导出未实现。
- 设置页中的黑名单编辑未实现。
- 设置页中的黑名单编辑未实现。
- 配置导入导出之外的“云端同步配置与数据”只完成配置快照手动推送，页面、会话、删除传播和远端合并未实现。
- GitHub 项目页打开、设置页打开、历史页打开、复制提取内容、独立重新提取按钮等侧边栏顶部动作未实现。
- 提取区拖拽调高、输入区拖拽调高、图片预览移除等交互未实现。

### 3.3 未对齐

- 文档声明“本文档只描述当前已经实现的产品能力”，但文中大量能力当前并未实现，文档口径已经超前。
- 文档宣称主要入口包含“对话管理页”，但代码里的 `conversations` 仍是 `PageShell` 占位，不具备真实业务能力。
- 文档宣称首次安装打开“快速上手文档”，当前代码实际打开的是内部占位式 `welcome` 页面，不是成型的快速上手文档。
- 文档宣称侧边栏顶部已实现 `复制提取内容 / 重新提取 / 清空页面 / 打开设置 / 打开 GitHub` 等控件，当前代码只有 `Readability`、`Jina` 和 `browserTab` 标识。
- 文档宣称“支持基于当前页面内容发起对话”，但当前发送链路没有把页面正文真正拼入 LLM 请求；`includePageContent` 仅存在于 UI 和命令参数中，调度层未使用。
- 文档宣称“支持导出会话”，当前 UI 有导出按钮，但后台没有接入导出实现，实际不可用。
- 文档宣称“支持导入导出配置、云端同步配置与数据”，当前已实现配置快照的手动同步，但不是全量数据同步服务。

- `docs/Workspace/sidebar.md` 中“阶段 4 实际交付”的大方向基本成立：最小顶部区、提取区、单 `Chat` 工作区、主聊天流恢复。

- `docs/Services/logger.md`、`docs/DataSchema/*`、`docs/dao/config-repository.md`、`docs/dao/page-repository.md`、`docs/dao/conversation-repository.md` 与当前域模型和仓储结构大体一致。

- `docs/Services/runtime-messaging.md` 把 `EXPORT_CONVERSATION` 放进命令集合，但后台没有注入 `conversationExporter`，该命令当前实际不可用。
- `docs/browser-entry.md` 与 `prd_docs` 一样，把首次安装行为表述为打开快速上手文档；当前代码是打开内部 welcome 占位页。

## 5. 现状分层判断

从第一性原理看，当前系统真正稳定的主链路只有：

1. 进入扩展。
2. 获取当前网页内容。
3. 在单一 `Chat` 上发起一次主回答流。
4. 把基础配置存到本地。

其余大量产品闭环还停留在：

1. 只有 schema，没有业务编排。
2. 只有文档，没有源码。
3. 只有 UI 占位，没有后台命令。
4. 只有按钮，没有实际能力。

## 6. 建议的文档修正顺序

1. 先修 `prd_docs/product-functional-spec.md`，把“当前已实现”改成“当前代码已实现 + 规划中”，避免继续误导。
2. 再修 `/docs` 顶层文档，把“目标架构文档”和“当前交付状态文档”分开。
3. 对 `docs/Workspace/settings.md`、`docs/Workspace/sidebar.md`、`docs/Services/runtime-messaging.md` 做一次逐项收口，删掉当前代码没有的“已实现口径”。
4. 如果后续要继续开发，优先补齐对话管理页、快捷输入多标签、会话导出、页面/标签清理，再考虑同步和分支能力。
